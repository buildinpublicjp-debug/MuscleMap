# MuscleMap — 画面設計ドキュメント

> **目的:** 各画面が「なぜ存在するか」「ユーザーがその画面を開いた時の感情・文脈」を言語化する。
> 新画面の実装・既存画面の修正時に必ず参照すること。
> UI実装の判断基準はCLAUDE.mdのUIデザイン原則を参照。

---

## 画面遷移マップ

```
起動
 └── オンボーディング（初回のみ）
      └── [完了] → MainTabView

MainTabView（5タブ）
 ├── [Tab 0] ホーム
 │    ├── → 分析メニュー（sheet）
 │    │    ├── → 週次サマリー
 │    │    ├── → バランス診断
 │    │    ├── → 筋肉ジャーニー
 │    │    └── → ヒートマップ
 │    └── 回復マップ / Strength Map 切替（ホーム内）
 │         └── [Pro未加入] → Paywall（sheet）
 │
 ├── [Tab 1] ワークアウト開始
 │    └── → ワークアウト実行中（fullScreenCover）
 │         └── → ワークアウト完了（画面内遷移）
 │
 ├── [Tab 2] 種目辞典
 │    └── → 種目詳細（sheet）
 │
 ├── [Tab 3] 履歴
 │    ├── マップ表示 / カレンダー表示 切替
 │    └── [マップ表示] 筋肉タップ → 部位詳細（halfModal）
 │         └── → 3D詳細（push）
 │
 └── [Tab 4] 設定
      └── サブスクリプション管理 → Paywall（sheet）
```

## 各画面の詳細

---

### オンボーディング（7画面）

**ファイル:** `Views/Onboarding/`

**存在理由:**
「なんのアプリかわからない」まま使い始めると離脱する。最初の30秒でコンセプトを体感させる。

**ユーザーの感情推移:**
1. 「また普通のトレーニングアプリか」（懐疑）
2. 「筋肉マップ？ちょっとおもしろそう」（興味）
3. 「体重とトレーニング頻度を入れると…」（参加意識）
4. 「自分の筋肉が動いてる」（インタラクティブデモで没入）
5. 「これ使ってみたい」（完了モチベーション）

**絶対条件:**
- インタラクティブデモ（筋肉をタップして光る）を必ず含める。テキスト説明だけでは弱い
- スキップ可能にする（強制は離脱を招く）
- 最後の画面で「始める」ボタンを押した瞬間に本体へ移行

---

### ホーム（Tab 0）

**ファイル:** `Views/Home/HomeView.swift` とその配下

**存在理由:**
ジムに着いた瞬間に開く画面。「今日どこ鍛えればいい？」への即答が唯一の使命。

**ユーザーの感情・文脈:**
- ロッカーでスマホを開く（時間的プレッシャーあり）
- 「腕昨日やったっけ」という不安を解消したい
- 提案されたメニューを見て「よし今日はこれで行こう」と決断したい

**画面内の2モード:**

#### 回復マップモード（デフォルト・無料）
- 筋肉の色で回復状態を一目で判断できる
- 緑=いける、赤=まだ早い、紫点滅=放置しすぎ
- 「今日のおすすめ」カードが筋肉マップと同等の存在感で表示される
- ストリーク（連続記録日数）が継続モチベーションを維持

#### Strength Mapモード（Pro）
- PRデータから筋肉の「太さ」で発達レベルを可視化
- 初期は全部細い → 「育てたい」欲求を生む
- PR更新のたびに太くなる → 「育った」実感
- シェアカードとして使える → SNSでのflex

**ユーザーの意思決定フロー（回復マップ）:**
```
画面を開く（3秒以内に判断したい）
 ↓
[緑の筋肉多い] → 「どこでも行ける。提案に従おう」
 ↓
[赤の筋肉多い] → 「昨日追い込んだ。提案を見てみよう」
 ↓
今日のおすすめを確認 → ワークアウト開始 → Tab 1へ
```

**絶対条件:**
- 人体図は頭から足先まで完全に表示（切れ禁止）
- 「今日のおすすめ」はスクロールなしで見える位置に配置
- 回復マップ ↔ Strength Map の切替ボタンは目立つが邪魔にならない位置

### ワークアウト開始（Tab 1）

**ファイル:** `Views/Workout/WorkoutStartView.swift`

**存在理由:**
「今日のメニューが決まった。記録を始めたい」という状態で来る。
最小タップでセッションを開始させる。

**ユーザーの感情・文脈:**
- すでにトレーニングを始めようとしている（行動意欲が高い）
- 迷わせてはいけない。種目選択・開始まで3タップ以内が理想
- 前回の記録が見えると「前回75kgだった。今日は77.5kgで行こう」と判断できる

---

### ワークアウト実行中（fullScreenCover）

**ファイル:** `Views/Workout/` 配下の実行系ファイル

**存在理由:**
セットを1つ1つ記録する。数字を入れることに集中させる。

**ユーザーの感情・文脈:**
- 筋肉が張っている。汗をかいている。
- 「早く次のセットに行きたい」。余計なUIは邪魔
- レストタイマーが動いている間は次のセットを「待っている」状態
- PR更新が起きた瞬間は高揚感MAX → ここで祝福演出が刺さる

**絶対条件:**
- セット完了は1タップ + Haptic (medium)
- PR更新 → 紙吹雪アニメーション + Haptic (heavy) + 祝福モーダル
- レストタイマーはカウントダウン形式（残り時間が見える）

---

### ワークアウト完了

**ファイル:** `Views/Workout/WorkoutCompletionView.swift`

**存在理由:**
「お疲れ様」の体験。今日のトレーニングの成果を可視化して達成感を与える。

**ユーザーの感情・文脈:**
- 達成感と疲労感が混在している
- 「今日どんなセッションだったか」を数字で確認したい
- シェアしたくなるような見た目が重要（SNSでのflex起点）

**絶対条件:**
- 刺激した筋肉がビジュアルで表示される（何を鍛えたかが一目でわかる）
- 合計セット数・重量・PR更新があったかどうかを表示
- シェアボタン（ワークアウトサマリーカードとして書き出せる）

---

### 種目辞典（Tab 2）

**ファイル:** `Views/Exercise/`

**存在理由:**
「この種目ってどこに効くの？」「肩を鍛える種目って何がある？」に答える。
トレーニング前後に「勉強」する用途。

**ユーザーの感情・文脈:**
- 「最近マンネリしてきた。新しい種目を探したい」
- 「今日のメニューにこの種目が来たけどどこに効くの？」
- 緊急性は低い。じっくり見る用途

**絶対条件:**
- 各種目にGIFアニメーション（正しいフォームが直感的にわかる）
- 刺激する筋肉をミニ人体図でハイライト
- 部位でフィルタリングできる（「肩だけ表示」等）

---

### 履歴（Tab 3）

**ファイル:** `Views/History/`

**存在理由:**
「最近ちゃんと鍛えられてるか？」「バランスが偏ってないか？」を確認する。
週単位・月単位の振り返り用途。

**ユーザーの感情・文脈:**
- トレーニング後または就寝前に確認する
- 「胸ばっかり鍛えて背中サボってたな」という気づきが欲しい
- カレンダーで「今週4回行けた」という継続実感を得たい

**2つの表示モード:**

#### マップ表示
- 期間内に刺激した筋肉のハイライト
- 筋肉をタップ → 部位詳細ハーフモーダル

#### カレンダー表示
- トレーニングした日が色付きで表示
- 連続記録の可視化

**部位詳細ハーフモーダル:**
- 「この部位、最近どれくらい鍛えた？何の種目で？」に答える
- `.medium`状態でスクロールなしに最重要情報が見える
- 3Dビジュアルは該当部位にズーム（全身表示しない）

---

### 分析メニュー（sheet）

**ファイル:** `Views/Home/AnalyticsMenuView.swift` とその配下

**存在理由:**
ホーム画面の「今日の判断」とは別軸で、「傾向・パターン」を把握するための画面群。
将来的にPro機能としてゲート化予定。

**4つのサブ画面:**

| 画面 | 目的 | ユーザーの問い |
|:--|:--|:--|
| 週次サマリー | 直近1週間の振り返り | 「今週ちゃんとできた？」 |
| バランス診断 | 部位偏りの可視化 | 「どこが弱点？」 |
| 筋肉ジャーニー | 長期的な変化の追跡 | 「半年でどう変わった？」 |
| ヒートマップ | 時間帯・曜日パターン | 「いつ行くのが自分のパターン？」 |

---

### Paywall（sheet）

**ファイル:** `Views/Paywall/PaywallView.swift`

**存在理由:**
Pro機能への入口。「もっと使いたい」という欲求が最大になった瞬間に表示する。

**ユーザーの感情・文脈:**
- 無料機能で「価値」を感じた後に来る（ここ重要）
- Strength Mapを見て「これの完全版が見たい」という状態
- 迷わせずに「月¥590でこれが全部使える」を瞬時に伝える

**表示タイミング（ゲートポイント）:**
- HomeViewでStrength Mapボタンをタップ（Pro未加入時）
- SettingsViewのサブスクリプション管理

**絶対条件:**
- 「無料でできること」vs「Proでできること」の対比を視覚的に見せる
- 年額プランを推奨表示（¥4,900/年 = 月¥408換算で目立たせる）
- 「まず試す」ではなく「これは自分に必要」と思わせるコピー

---

### 設定（Tab 4）

**ファイル:** `Views/Settings/`

**存在理由:**
プロフィール設定・課金管理・言語設定・法的文書へのアクセス。
積極的に使う画面ではなく、必要な時だけ開く。

**ユーザーの感情・文脈:**
- 体重を変更したい（Strength Map計算に影響する）
- 課金状況を確認したい
- 言語を変えたい

## 画面実装時のチェックリスト

新しい画面または既存画面を大幅修正した際に必ず確認：

- [ ] この画面を開くユーザーの**感情・文脈**は何か？（上記ドキュメントを参照）
- [ ] その感情に対して、画面が**3秒以内**に答えを出せているか？
- [ ] **人体図・ビジュアル要素**が切れていないか？（スクショ確認）
- [ ] **最重要情報**がL3以上のフォントで表示されているか？
- [ ] **Hapticフィードバック**が実装されているか？
- [ ] **同じデータを表示する他画面**と品質が揃っているか？

# MuscleMap — 画面設計分析プロンプト（Agent Teams用）

> このプロンプトをClaude Code / Agent Teamsに渡して、
> 「新画面の提案」「既存遷移の改善提案」を分析させる。

---

## 📋 使い方

```
以下のプロンプト全文をコピーして、Claude Codeまたはエージェントに渡す。
事前に以下のファイルを読み込ませること：
  - docs/SCREENS.md
  - docs/SCREEN_FLOWS.md
  - CLAUDE.md
```

---

## プロンプト本文

---

あなたはiOSアプリ「MuscleMap」のUX設計アナリストです。
以下のドキュメントを読んだ上で、分析を実施してください。

**読み込むファイル:**
- `docs/SCREENS.md` — 全画面の存在目的・ユーザー感情・要件
- `docs/SCREEN_FLOWS.md` — 全画面遷移の条件とトリガー
- `CLAUDE.md` — アプリの設計思想・カラー・UIルール

---

## 分析タスク

### Task 1: 既存遷移のUXギャップ分析

以下の観点で、現在の画面遷移における「ユーザーが詰まりやすいポイント」「感情が途切れるポイント」を特定してください。

**チェックポイント:**
1. **離脱リスクが高い遷移** — ユーザーが「次に何をすればいいか」迷う可能性がある遷移はどこか？
2. **感情のピークを活かせていない遷移** — 高揚感や達成感が高い瞬間（例: PR更新後、ワークアウト完了後）に、次の行動への誘導が弱い箇所はどこか？
3. **タップ数が多すぎる遷移** — 目的の画面まで到達するのに3タップ以上かかる重要フローはどれか？
4. **行き止まりになっている遷移** — dismiss後に「で、次は？」となってしまう画面はどれか？

**出力形式:**
```
## UXギャップ一覧

| # | 場所 | 問題 | ユーザーの感情状態 | 重要度（高/中/低） |
|---|------|------|-------------------|-------------------|
| 1 | xxx | xxx  | xxx               | 高                |
```

### Task 2: 新画面の提案

以下の3つの軸で、追加すべき画面・機能を提案してください。

**軸A: リテンション強化（継続して使ってもらうための画面）**
- 「また明日も開きたい」と思わせる仕掛けとして、どんな画面が有効か？
- 参考: ストリーク表示は既存。それ以上の動機づけができる画面は何か？

**軸B: 課金転換率向上（無料→Pro）**
- 無料ユーザーが「これはお金払う価値がある」と判断する瞬間を最大化する画面は何か？
- 現在のPaywallトリガーは HomeView の Strength Mapボタンのみ。他に有効なトリガーポイントはあるか？

**軸C: SNS拡散（バイラル）**
- 「Xに投稿したい」「友達に見せたい」という衝動が生まれる画面・瞬間を増やすには？
- 現在のシェアポイント: ワークアウト完了カード、Strength Mapシェアカード

**出力形式:**
```

### [軸A] リテンション強化

- 目的:
- 表示タイミング:

- 実装難易度（高/中/低）:
- 想定効果:

---

### [軸B] 課金転換率向上
（同形式）

### [軸C] SNS拡散
（同形式）
```

### Task 3: 遷移パターンの改善提案

以下の具体的な遷移について、改善案を考えてください。

**3-1. ワークアウト完了 → 次の行動**
現状: WorkoutCompletionView に「ホームへ戻る」ボタンのみ。
問い: 達成感が最大のこの瞬間に、何を見せると継続率・課金率が上がるか？

**3-2. オンボーディング完了 → ホーム**
現状: 完了後すぐにホーム画面（筋肉マップ）が表示される。
問い: 初めてホーム画面を見たユーザーが「次何をすればいい？」と迷わないために、何が必要か？

**3-3. Pro機能の「チラ見せ」タイミング**
現状: Paywallは Strength Mapボタンタップ時のみ。
問い: 無料ユーザーにProの価値を感じさせる「チラ見せ」遷移を、どこに・どのように挿入するのが最も効果的か？ユーザーの不快感を最小化しつつ認知させる方法を提案せよ。

**出力形式:**
```
## 遷移改善提案

### 3-1. ワークアウト完了 → 次の行動
現状の問題:
改善案:
実装方法の概要:
期待される効果:

（以下同形式）
```

### Task 4: 優先度マトリクス

Task 2・3で提案した内容を、以下のマトリクスで整理してください。

```
高インパクト × 低コスト → 最優先（今すぐやる）
高インパクト × 高コスト → 次フェーズ（設計してから）
低インパクト × 低コスト → 余裕があれば
低インパクト × 高コスト → やらない
```

**出力形式:**
```
## 優先度マトリクス

### 🔥 最優先（今すぐやる）
- [ ] xxx（リテンション / 難易度: 低 / 期待効果: xxx）

### 📋 次フェーズ
- [ ] xxx

### 💡 余裕があれば
- [ ] xxx

### ❌ やらない
- xxx（理由: xxx）
```

## 分析時の制約・前提

以下を踏まえた上で提案すること:

1. **サーバーレス制約（現時点）**
   全データは端末内のSwiftDataに保存。Supabase等のサーバーサイド実装が必要な提案は「将来フェーズ」として明記すること。

2. **RevenueCat未接続**
   現在 `isPremium = true` のハードコード。課金フローの提案はUI設計に留め、SDK接続の話はスコープ外。

3. **デザイントーン**
   「バイオモニター × G-SHOCK」のダーク系。過度にポップ・カジュアルなUI提案はCLAUDE.mdのトーンに反する。

4. **ユーザー像**
   週3〜5回ジムに通う20-35歳。記録を続けたいが面倒くさがりな側面もある。SNSへの発信意欲あり（特にX）。

5. **既存の強み（活かすこと）**
   - 筋肉マップの視覚的インパクト
   - PRデータの蓄積（Strength Mapの基盤）
   - ワークアウト完了時の達成感演出（PR祝福モーダル）

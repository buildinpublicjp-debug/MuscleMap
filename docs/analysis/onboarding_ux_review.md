# オンボーディングUXレビュー — 2026-03-21

> 対象: SplashView + OnboardingV2View（9ページ）+ NotificationPermissionView
> レビュアー: Claude Code
> バージョン: v1.1（commit 19e370e 以降）

---

## 全体評価

- **総合スコア: 7.5 / 10**
- **最大の問題:** 情報量過多 — RoutineBuilderPage（1078行）は小さな画面にDay別location切替・種目入替・セット数編集を詰め込みすぎ。初回ユーザーが操作方法を理解できない可能性が高い
- **最も良い点:** 筋肉マップの一貫した活用 — 全ページで筋肉マップを使い、ユーザーの選択がリアルタイムに反映される体験は非常に直感的。目標選択→分割法→種目選択が筋肉マップのハイライトで繋がっている

### 全体的な強み
1. **ビジュアルファースト設計**: GIF・筋肉マップ・アニメーションが豊富で、フィットネスアプリとして視覚的訴求力が高い
2. **段階的パーソナライズ**: 目標→頻度→場所→経験と段階を踏んでルーティンを構築する流れが論理的
3. **一貫したデザイン言語**: mmOnboarding カラーパレット（#1A1A1E bg / #00E676 accent）が全ページ統一
4. **スキップ可能設計**: PRInputPageはスキップ可能、通知許可は「あとで」選択肢あり

### 全体的な弱み
1. **ページ数過多**: 最大9ページは平均的なフィットネスアプリの2倍。離脱リスクが高い
2. **超回復アニメーションの重複**: FrequencySelectionPage / GoalMusclePreviewPage / NotificationPermissionView の3か所で類似アニメーションを使用。初回ユーザーへの説明効果は1回目で十分
3. **戻り操作の不在**: TabViewスワイプで戻れるが、明示的な「戻る」ボタンがない。ユーザーが前のページの選択を修正したい場合に不安
4. **進捗の非永続化**: force-quit後にPage 0からやり直し（currentPageが@Stateのみ）

## ページ別レビュー

### Page 0: SplashView（スプラッシュ）

**ファイル:** `Views/Onboarding/SplashView.swift` (246行)

**良い点:**
- 筋肉マップのウェーブアニメーションがアプリの核心機能を即座に伝える
- タイムラインが適切: タイトル(0s) → マップ(0.3s) → タグライン(0.8s) → ボタン(1.5s)
- ボタンの脈動グローが適度な urgency を生む
- タグライン「鍛えた筋肉が光る。」が簡潔で記憶に残る

**問題点:**
- P1: ウェーブアニメーションのループ（全点灯→2秒ホールド→リセット→1秒待ち→再開）は、ユーザーがボタンに気づく前に全筋肉が消灯して「壊れた？」と感じる可能性がある
- P2: `Spacer().frame(height: 60)` が小さい端末（iPhone SE/Mini）で上部余白を取りすぎる可能性
- P3: 「始める」ボタンまで1.5秒待たせるのは、再訪ユーザー（force-quit後再起動）にはストレス

**改善案:**
1. ウェーブアニメーションの全消灯をやめ、最低限の筋肉は常時薄く点灯（0.2 opacity）で存在感を維持
2. `Spacer().frame(height: 60)` → `Spacer(minLength: 20).frame(maxHeight: 60)` で小画面対応
3. 2回目以降の訪問時はアニメーションを短縮（AppStateフラグで判定）

**優先度:** Low

### Page 1: GoalSelectionPage（目標選択）

**ファイル:** `Views/Onboarding/GoalSelectionPage.swift` (429行)

**良い点:**
- スライダーUIでの優先度設定は、単純なタップ選択より詳細なパーソナライズが可能
- スライダー値に連動して筋肉マップがリアルタイムにハイライトされる体験が秀逸
- 7目標のコピーがエモーショナル（「舐められたくない」「モテたい」）で日本語圏のターゲットに刺さる
- 重点部位チップが横スクロールで選択結果をフィードバック

**問題点:**
- P1: **スライダーUIの学習コスト** — 「スライダーを右にドラッグ」というヒントはあるが、7つのスライダーを初見で理解・操作するのは認知負荷が高い。Hevy/Strong等の競合はタップ選択で済ませている
- P2: **筋肉マップ260ptが大きすぎ** — 7つのスライダーカードをScrollViewに入れているが、マップが画面の半分近くを占めるため、カードが2-3個しか見えず下にスクロールが必要
- P2: **goalValues の初期状態** — force-quit後に再訪した場合、AppState.shared.userProfile.goalWeights に保存済みデータがあるが、goalValues@State は空で再開。保存済みデータとの不整合が発生（audit H-5で指摘済み）
- P3: スライダーのステップ0.05（20段階）は細かすぎる。ユーザーは「やや重要」「とても重要」程度の粒度で十分

**改善案:**
1. スライダーを3段階（0/0.5/1.0 = なし/やや/とても）に簡略化。または「タップで選択 + 長押しで優先度UP」パターン
2. 筋肉マップの高さを200ptに縮小し、カードの可視領域を確保
3. `onAppear` で `AppState.shared.userProfile.goalWeights` を読み込んで `goalValues` を復元
4. スライダーの代わりにタップトグル（0→0.5→1.0→0 のサイクル）を検討

**優先度:** High

### Page 2: FrequencySelectionPage（頻度選択）

**ファイル:** `Views/Onboarding/FrequencySelectionPage.swift` (594行)

**良い点:**
- 超回復アニメーションが「頻度＝回復サイクル」の関係を視覚的に教育
- タイムラインバーが曜日別スケジュールを一目で把握可能
- 「初心者におすすめ」バッジが週2回についており、迷うユーザーを誘導
- 医学的根拠テキスト（evidenceSection）が選択の確信度を高める

**問題点:**
- P2: **超回復アニメーションが速すぎる可能性** — 1.5秒/曜日で7日＝10.5秒1サイクル。色変化（赤→黄→グレー）の意味を理解する前にサイクルが終わる
- P2: **筋肉マップ320pt→240ptの突然の縮小** — 選択前320pt、選択後240ptへ縮小するが、スクロール位置がジャンプして混乱する可能性
- P3: **週6回の選択肢がない** — `fivePlus = 5` で「週5回以上」とラベルしているが、実際のtrainingDaysは月〜金の5日。週6-7回トレーニングしたいユーザーの期待と合わない
- P3: カラーレジェンド「刺激/回復中/未刺激」に「回復完了（緑）」がない。アニメーションでは回復完了筋肉がinactive（暗い）に戻るため、「回復＝消える」と誤解される可能性

**改善案:**
1. アニメーション速度を2.0秒/曜日に落とし、色変化の説明テキストをツールチップで表示
2. 筋肉マップの高さを選択前後とも260ptに固定し、レイアウトジャンプを防止
3. レジェンドに「✅ 回復完了」を追加（緑→暗いの遷移を明示）
4. `fivePlus` を `fiveSix = 5` にリネームし、週6回のtrainingDaysパターンも用意

**優先度:** Medium

### Page 3: LocationSelectionPage（場所選択）

**ファイル:** `Views/Onboarding/LocationSelectionPage.swift` (432行)

**良い点:**
- GIFマーキーギャラリーが場所選択の結果（使える種目）を直感的にフィードバック
- 種目数バッジ（`exerciseCountLabel`）が定量的な差異を示す
- タップで種目詳細が見れる（`ExerciseGifView` → `ExerciseDetailView`）
- 4選択肢が明確に分かれており、迷いが少ない

**問題点:**
- P1: **マーキーギャラリーのパフォーマンス** — `TimelineView(.animation)` で毎フレーム再描画。GIF 20枚×2行×2セット＝80個のGIFを常時レンダリングしているため、低スペック端末でフレームドロップの可能性
- P2: **「両方」の定義が曖昧** — `.both` は equipmentFilter が `["バーベル", "ダンベル", "自重"]` だが、ユーザーは「ジムの日と自宅の日を混ぜる」と理解する可能性がある。RoutineBuilderPageでのDay別location切替との関係が不明
- P3: **選択前のデフォルト表示** — `selected == nil` の状態でGIFギャラリーは `.gym` と同じ全種目を表示（`case .gym, .both, .none:`）。選択前に全種目が見えると「場所で変わる」メリットが伝わらない

**改善案:**
1. マーキーギャラリーのGIF数を10枚×2行に削減（prefix(10)）。または `LazyHStack` + `ScrollView` に変更してレンダリング負荷を軽減
2. `.both` の subtitle を「ジムの日と自宅の日を組み合わせ（Day別に場所を選択可能）」に変更
3. 選択前はGIFギャラリーを非表示にし、選択後にアニメーション表示（差異を強調）

**優先度:** Medium

### Page 4: ProfileInputPage（プロフィール入力）

**ファイル:** `Views/Onboarding/TrainingHistoryPage.swift` (616行)

**良い点:**
- 4要素（経験・身長・体重・ニックネーム）を1ページに統合し、ページ数削減に貢献
- 経験レベルに応じた筋肉マップ表示（beginner=全暗い、veteran=全身色付き）が直感的
- 体重ステッパーの kg/lb 切替が海外ユーザー対応
- BMI・体脂肪率カテゴリのリアルタイム表示がフィードバック豊富
- 体脂肪率がON/OFFトグルで任意入力

**問題点:**
- P1: **入力項目が多すぎる** — 経験+身長+体重+体脂肪率+ニックネームの5項目をスクロールで操作。特にステッパー3つ（身長・体重・体脂肪率）が連続するのは単調
- P2: **長押しジェスチャーのDiscoverability** — 身長ステッパーの長押しで±5cm移動する機能が隠れている。140cm→180cmまで40回タップが必要で、長押しを知らないと離脱しやすい
- P2: **ニックネームのキーボード問題** — TextFieldにフォーカスするとキーボードが画面を覆い、「次へ」ボタンが見えなくなる。ScrollViewはあるが、自動スクロールしない
- P3: **筋肉マップが100ptで小さすぎ** — 経験レベルの差異（beginner vs veteran）が視覚的にわかりにくい

**改善案:**
1. 身長・体重をスライダーまたはPickerに変更し、操作回数を削減。または Wheel Picker で直感的に設定
2. 長押しの存在を示すヒント（初回のみツールチップ: 「長押しで高速移動」）を追加
3. `.scrollDismissesKeyboard(.interactively)` を追加し、キーボード表示時の操作性改善
4. 筋肉マップを160ptに拡大し、経験レベル差異を明確化

**優先度:** High

### Page 5: PRInputPage（PR入力 — 経験者のみ）

**ファイル:** `Views/Onboarding/PRInputPage.swift` (767行)

**良い点:**
- BIG3デフォルト表示で「何を入力すればいいか」が明確
- 筋肉マップタップ→種目選択→重量入力の3ステップフローが構造化されている
- 強さレベルプレビュー（D〜S）がゲーミフィケーション要素として魅力的
- 「わからない場合はスキップ →」で離脱防止
- 経験者のみ表示（beginner/halfYearはスキップ）が適切なターゲティング

**問題点:**
- P1: **3段階シート遷移の複雑さ** — 筋肉タップ→MuscleExerciseSheet→WeightInputSheet のフローで、ユーザーは2回のシート表示を経由する。シート→シートの遷移で現在地を見失いやすい
- P2: **重量入力の初期値が0kg** — `initialWeight ?? 0` のため、新規入力時に0kgから始まる。BIG3なら体重ベースのデフォルト値（例: ベンチプレス → 体重×0.7）を提案すべき
- P2: **長押し速度（0.12秒/0.25kg増分）** — 0kg→100kgまで長押しで48秒かかる。実用的でない
- P3: **340ptの筋肉マップが大きすぎ** — PR入力の主役は「種目選択+重量入力」であり、マップは補助的。220ptで十分

**改善案:**
1. MuscleExerciseSheet と WeightInputSheet を統合し、種目選択→即重量入力の1ステップフローに
2. 重量の初期値を体重ベースで推定（ベンチプレス: `bodyweightKg * 0.7`、スクワット: `bodyweightKg * 1.0`、デッドリフト: `bodyweightKg * 1.2`）
3. 長押し増分を 2.5kg/0.12秒 に変更（0→100kgが約5秒）
4. 筋肉マップを220-260ptに縮小

**優先度:** Medium

### Page 7: RoutineBuilderPage（ルーティン構築）

**ファイル:** `Views/Onboarding/RoutineBuilderPage.swift` (1078行)

**良い点:**
- Day別タブバーで複数日のルーティンを1画面で確認・編集可能
- 種目の入替・追加・削除が可能で、ユーザーに最終決定権がある
- セット数×レップ数の編集がインラインで完結
- 自動生成ロジック（目標筋肉優先度→器具フィルタ→上位3-4種目）が妥当
- 教育ヒント（筋肉グループ別のトレーニングTips）が付加価値

**問題点:**
- P1: **認知負荷が極めて高い** — Day別location切替、種目タップで入替/追加シート、セット×レップ編集、GIFプレビュー、教育ヒントが1画面に集約。初回オンボーディングでこのレベルの編集UIを提示するのは過剰
- P1: **1078行のView** — CLAUDE.md の200行制限を大幅に超過。メンテナンス困難
- P2: **「このメニューで始める」vs「自分でDay別にカスタマイズ」の2ボタン** — CTAが2つあると迷う。大多数のユーザーは自動生成のまま進みたい
- P2: **Day別location切替の意味が不明** — LocationSelectionPageで場所を選んだにもかかわらず、Day別に場所を変更できる。一貫性がない
- P3: **種目入替シートがフルスクリーン** — 入替対象の種目を覚えたまま新種目を選ぶ必要がある。元の種目が見えなくなる

**改善案:**
1. **オンボーディングではread-onlyプレビューに限定し、編集機能は設定画面のRoutineEditViewに委ねる**。「このメニューで始める」ボタンのみ
2. Viewを分割: RoutineBuilderPage（コンテナ、~200行） + RoutineDayTab（タブ、~150行） + RoutineExerciseGrid（グリッド、~150行） + RoutineExercisePicker（シート、~200行）
3. Day別location切替を削除。LocationSelectionPageで選んだ場所を全Dayに適用
4. 種目入替シートを half modal にし、元の種目をスクロールで確認可能に

**優先度:** Critical

### Page 8: NotificationPermissionView（通知許可）

**ファイル:** `Views/Onboarding/NotificationPermissionView.swift` (275行)

**良い点:**
- 「回復したら、教える。」のコピーが通知の価値を端的に伝える
- 成長サイクル3ステップバッジ（刺激→回復→成長）が教育的
- 通知プレビューカード2枚がiOS標準の見た目で、ユーザーが受け取る通知をイメージしやすい
- 「あとで」スキップオプションがAppleのHIGに準拠
- 筋肉マップの4段階回復アニメーションが通知の必要性を視覚的に伝える

**問題点:**
- P2: **回復アニメーション1.2秒/フェーズが速い** — 赤→黄→緑→暗い の4段階を4.8秒でサイクル。「回復」のプロセスを実感するには速すぎる
- P2: **「回復通知をオンにする」ボタンがPaywallっぽい** — グリーングラデーションの大きなボタンが、通知許可なのに課金ボタンと誤認される可能性
- P3: **通知拒否後のリカバリーなし** — 「あとで」を選んだ後、アプリ内で再度通知許可を促す導線がない
- P3: **DispatchQueue.main.async の使用** — `requestAuthorization` のコールバックで `DispatchQueue.main.async` を使用しているが、`@MainActor` メソッド内なので `Task { @MainActor in }` パターンの方が一貫性がある

**改善案:**
1. アニメーション速度を2.0秒/フェーズに落とし、各フェーズでテキストラベル（「刺激！」「回復中...」「回復完了！」）を表示
2. ボタンの色をシステムブルー or 白背景に変更し、Paywallとの差別化
3. 設定画面に「通知を有効にする」誘導バナーを追加（拒否ユーザー向け）

**優先度:** Low

### Page 9: RoutineCompletionPage（完了 + ハードペイウォール）

**ファイル:** `Views/Onboarding/RoutineCompletionPage.swift` (491行)

**良い点:**
- 順次ハイライトアニメーション（0.8秒/種目）で「自分のルーティン」を振り返る体験
- 4カラム70×70 GIFグリッドがコンパクトで一覧性が高い
- カバー率+プログレスバーが達成感を演出
- 目標連動キャッチコピー（「デカくなる準備、できた。」）がエモーショナル
- 「無料ではじめる」が明示的で、強制課金ではないことが伝わる

**問題点:**
- P1: **ハードペイウォール位置が最後** — 9ページのオンボーディングを完了してからPaywall表示は、ユーザーが「全部無料で使える」と思い込んだ後に壁にぶつかる体験。購買意欲が最も高い瞬間（ルーティン完成時）にPaywallを出すのは戦略的だが、離脱時の失望感も大きい
- P2: **GIFアニメーション中にスクロール不可** — 順次ハイライト中（isAnimating=true）にScrollViewが固定される可能性。種目数が多い場合、画面外の種目が見えない
- P2: **PaywallView の fullScreenCover が遷移先不明** — crownボタンタップで PaywallView(isHardPaywall: true) が表示されるが、ユーザーがPaywallを閉じた場合の挙動が不明確
- P3: **法的リンク（利用規約・プライバシーポリシー）が小さすぎ** — font(.system(size: 10)) では見つけにくい

**改善案:**
1. Paywall前に「Pro機能プレビュー」セクション（Strength Map・無制限ワークアウト等）を追加し、有料の価値を事前に伝える
2. アニメーション完了後にスクロールを自動的に有効化
3. PaywallView 閉じた後の状態を明確に（「無料ではじめる」ボタンにフォーカスを戻す）
4. 法的リンクを12-13ptに拡大し、App Store審査対応を確実に

**優先度:** High

## OnboardingV2View（コンテナ）

**ファイル:** `Views/Onboarding/OnboardingV2View.swift` (153行)

**良い点:**
- PRInputPageの条件分岐（経験者のみ表示）が適切
- ページインジケーターが動的（PRスキップ時はドット数変動）

- `.tabViewStyle(.page(indexDisplayMode: .never))` でカスタムインジケーターを使用

**問題点:**
- P1: **スワイプで自由にページ遷移可能** — TabViewの標準動作で、ユーザーが未入力のまま先のページに進める。例: 目標未選択のまま頻度選択ページに進むと、空のgoalPriorityMusclesでアニメーションが動作しない

**改善案:**
1. `TabView` のスワイプを無効化し、「次へ」ボタンでのみページ遷移可能にする（`.tabViewStyle(.page(indexDisplayMode: .never))` → カスタム遷移）
2. ページインジケーターの位置を調整（ボタンの上に十分なスペース確保）
3. Page 1以降に「< 戻る」ボタンを追加（左上またはインジケーター横）

**優先度:** High

## 横断的な問題

### 1. アニメーション速度の不統一

| ページ | アニメーション | 速度 |
|:---|:---|:---|
| FrequencySelection | 超回復サイクル | 1.5秒/曜日 |
| GoalMusclePreview | 超回復サイクル | 0.6秒/曜日 |
| NotificationPermission | 回復フェーズ | 1.2秒/フェーズ |

同じ「回復」概念のアニメーションが3種類の速度で動いている。**1.2-1.5秒/ステップに統一すべき。**

### 2. 筋肉マップの高さ不統一

| ページ | マップ高さ |
|:---|:---|
| GoalSelection | 260pt |
| Frequency | 240-320pt |
| Profile | 100pt |
| PRInput | 340pt |
| GoalMusclePreview | 300pt |
| Notification | 300pt |

ページごとに異なるサイズは一貫性に欠ける。**主要ページは280-300pt、補助ページは200pt に統一を推奨。**

### 3. isProceeding ガードの重複パターン

全ページで `@State private var isProceeding = false` + `guard !isProceeding else { return }` パターンを使用。onAppear で `isProceeding = false` にリセットしているが、共通プロトコルまたはViewModifier として抽出可能。

### 4. Haptic パターンの統一

全ページで `HapticManager.lightTap()` を使用しているが、「選択」「遷移」「エラー」で Haptic の種類を分けるべき:
- 選択: `.lightTap()` (現状)
- ページ遷移: `.medium()` (未実装)
- エラー/無効操作: `.error()` (未実装)

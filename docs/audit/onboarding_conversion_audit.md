# MuscleMap オンボーディング コンバージョン最適化監査

> **v1.0 | 2026-03-18**
> 対象: SplashView → OnboardingV2View (9ページ) → NotificationPermissionView → ハードPaywall
> 手法: ソースコード全量読解によるUX/コンバージョン分析（スクリーンショット撮影なし）

---

## Executive Summary

MuscleMapのオンボーディングは**Splash + 9ページ + 通知許可 = 最大11ステップ**の長尺フローで、ルーティンビルダー（Page 7）でサンクコストを蓄積し、完了ページ（Page 8）でハードPaywallへ誘導する設計。

**強み:**
- 目標選択（Page 0）が感情ドリブンで、筋肉マッププレビューが即座にパーソナライズ感を演出
- ルーティンビルダーのGIF付き種目選択がアプリ品質の高さを伝える
- GoalMusclePreviewPage（Page 5）がクライマックスとして感情のピークを作る

**致命的リスク:**
- **9ページは競合比較で2-3ページ多い**。Page 6（体重入力）とPage 3→4（トレ歴→PR入力）で大量離脱が予測される
- **ハードPaywallに閉じるボタンがない**が、RoutineCompletionPage側に「無料ではじめる」がある。この二重構造がUXの混乱を招く
- **PaywallViewに利用規約/プライバシーポリシーのリンクがない**（App Store審査リスク）
- **CallToActionPage.swiftがデッドコード**として残存（OnboardingV2Viewで参照なし）

**予測指標:**

| 指標 | 現状予測 | 最適化後予測 |
|:---|:---|:---|
| オンボーディング完走率 | 35-45% | 55-65% |
| Paywall表示到達率 | 30-40% | 50-60% |
| Trial開始率（対Paywall表示） | 12-18% | 20-30% |
| 全体コンバージョン率 | 4-7% | 10-18% |
| 推定MRR（1,000 DL/月） | ¥16,300-¥28,600 | ¥40,800-¥73,500 |

## 3. 感情アーク分析

```
感情レベル
  高 │                              ★ GoalMusclePreview
     │                             ╱  ╲
     │  ★ Splash               ╱     ╲         ★ RoutineCompletion
     │  ╱╲                   ╱        ╲       ╱
  中 │╱   ╲  Goal         ╱           ╲     ╱ RoutineBuilder
     │      ╲   ╱╲       ╱              ╲  ╱
     │       ╲╱   ╲    ╱                 ╲╱
  低 │    Freq  Loc  ╲╱                  Weight
     │           History PR
     └─────────────────────────────────────────→ ページ進行
      S   0   1   2   3   4   5   6   7   8   N
```

### 分析

**Phase 1: エントリ高揚 (S-0)**
- SplashViewの筋肉ウェーブアニメーション（前面6筋肉→背面6筋肉の順次点灯）が「このアプリは違う」感を演出
- GoalSelectionPageの感情コピー（「舐められたくない」「モテたい」）が個人の欲求に直結
- 筋肉マッププレビューが即座に「あなた専用」感を提供

**Phase 2: 情報収集の谷 (1-4)**
- 3ページ連続の質問形式（頻度→場所→歴歴）で感情が平坦化
- TrainingHistoryPage（Page 3）が感情の最低点。ビジュアル比率15%でテキスト偏重
- PRInputPage（Page 4）は入力摩擦が高いが、リアルタイムの筋力レベルバッジ表示が「データが活きる」感を維持

**Phase 3: クライマックス (5)**
- GoalMusclePreviewPageが感情の頂点。選択した目標×筋肉マップの大画面表示（220pt）
- 「打撃力は背中と脚から生まれる」等の目標別コピーが個人の欲求を具体的な筋肉に変換
- ここが「このアプリを使いたい」の最大瞬間

**Phase 4: 投資と回収 (6-8)**
- WeightInputPage（Page 6）で感情が再度低下。個人情報入力は義務感
- RoutineBuilderPage（Page 7）で「自分だけのメニューを組んでいる」感がサンクコストとして蓄積
- RoutineCompletionPage（Page 8）で「完成した！」の達成感とPaywallへの導線

**問題点:**
- Phase 2が3-4ページと長すぎ、「質問→質問→質問」の疲労感が蓄積
- GoalMusclePreview（Page 5）の後にWeightInput（Page 6）を置くのは感情設計として逆効果。クライマックスの直後に最も退屈な入力を要求している

## 4. サンクコスト効果分析

### 蓄積ポイントと効果

| ページ | 投資タイプ | サンクコスト強度 | 累積効果 |
|:---|:---|:---|:---|
| Page 0: Goal | 感情的自己開示 | ★★★☆☆ | 「自分の目標を選んだ」 |
| Page 1: Frequency | 行動計画 | ★★☆☆☆ | 「週の予定を決めた」 |
| Page 2: Location | 環境宣言 | ★★☆☆☆ | 「トレーニング環境を決めた」 |
| Page 3: History | 自己評価 | ★★☆☆☆ | 「自分のレベルを認めた」 |
| Page 4: PR | データ入力 | ★★★★☆ | 「自分の数値を入れた」 |
| Page 5: Preview | 感情的コミット | ★★★★★ | 「自分の目標筋肉を見た」 |
| Page 6: Weight | 個人情報 | ★★★☆☆ | 「体重まで入れた」 |
| Page 7: Routine | 時間+カスタマイズ | ★★★★★ | 「自分のメニューを組んだ」 |

**総合評価:**
- Page 7（RoutineBuilder）完了時のサンクコスト蓄積は非常に高い。GIF付き種目選択、Day分割、自動フィル+手動カスタマイズの組み合わせで「自分が時間をかけて作ったメニュー」という所有感が発生
- Page 8（RoutineCompletion）で完成ルーティンをサマリー表示し、「これを失いたくない」感情を喚起した上でPaywallを提示するのは理論的に正しい配置
- **ただし、Page 4-6で離脱した場合、この蓄積は無意味**。離脱率が最も高いのはPage 4（25%）とPage 6（20%）であり、RoutineBuilderに到達する前に失われる

### サンクコスト効果の限界

RoutineBuilderPage で組んだルーティンは `RoutineManager.shared.saveRoutine()` で端末に保存される。つまり**Paywall を閉じて無料で開始しても、ルーティンデータは利用可能**。ユーザーがこれに気づけばサンクコスト圧力は大幅に減少する。

### 評価

| 評価項目 | 評点 | 詳細 |
|:---|:---|:---|
| ヘッドライン訴求力 | B+ | 「あなた専用のプログラムを解放」は具体的だが、ベネフィットよりも機能寄り |
| ビジュアルインパクト | B | 筋肉マップ160ptは妥当だが、GoalMusclePreview(220pt)の後だと見劣り |
| 社会的証明 | F | レビュー、ユーザー数、事例が一切ない |
| 緊急性・希少性 | D | 「7日間無料」以外のアンカーなし。割引率の明示なし |
| 価格表示 | B | 年額の月換算（¥408）を表示している。月額との対比も明確 |
| CTAボタン設計 | A- | `mmAccentPrimary`で目立つ。ただし1つしかない（年額推奨のみ） |
| 法的準拠 | C- | 自動更新の説明はあるが、利用規約/プライバシーポリシーへのリンクがPaywall自体にない（RoutineCompletionPageにはある）。App Store審査基準3.1.2に抵触する可能性 |
| 閉じる導線 | 特殊 | isHardPaywall=trueで閉じるボタン非表示+interactiveDismissDisabled。ただしRoutineCompletionPage側の「無料ではじめる」で脱出可能 |

### ハードPaywallフロー分析

実際の遷移は以下:
1. RoutineCompletionPage表示 → 「Proを解放」タップ → `.fullScreenCover(PaywallView(isHardPaywall: true))`
2. PaywallViewで購入 → dismiss() → RoutineCompletionPageに戻る（ここからはonCompleteで通知許可へ）
3. PaywallViewで購入しない → PaywallViewを閉じられない（閉じるボタンなし+swipe down無効）
4. **唯一の脱出**: RoutineCompletionPage側の「無料ではじめる」ボタン（PaywallViewの裏側にある）

**問題: PaywallViewが`.fullScreenCover`で開かれるため、裏のRoutineCompletionPageは見えない。ユーザーが「無料ではじめる」に到達するには、PaywallViewの背後に戻る方法を知っている必要があるが、swipe downも閉じるボタンもない。**

→ **これは実質的にPaywallを閉じる手段がない状態であり、App Store審査で拒否されるリスクが極めて高い。** Apple HIG 4.0 は「ユーザーが課金なしでアプリの基本機能にアクセスできる道を確保すること」を要求している。

**修正案**: PaywallViewに`isHardPaywall`時でも表示される「無料で続ける」ボタンを追加するか、遅延表示（3-5秒後に閉じるボタンを表示）を実装する。

### 価格戦略分析

| プラン | 価格 | 月あたり | トライアル | 表示サイズ |
|:---|:---|:---|:---|:---|
| 年額（推奨） | ¥4,900/年 | ¥408/月 | 7日間無料 | 大（56pt高ボタン） |
| 月額 | ¥590/月 | ¥590/月 | なし | 小（テキストリンク風） |

- 年額/月額の価格差: 月額の約69%割引 → 十分な差だが、**割引率を明示していない**
- 年額のみ7日間トライアルがあるのは正しい戦略（高額プランにトライアルを付けてLTVを最大化）
- **月額にトライアルがないのは離脱要因**。月額を選ぶユーザーは慎重派であり、トライアルなしでは購入しない

| 特性 | MuscleMap | Strong | JEFIT | Hevy | Fitbod |

| 筋肉マップビジュアル | ★★★★★ | ★★☆☆☆ | ★★★☆☆ | ★☆☆☆☆ | ★★★☆☆ |

| 感情訴求 | ★★★★★ | ★★☆☆☆ | ★★☆☆☆ | ★★☆☆☆ | ★★★☆☆ |

### MuscleMapの競合優位性

1. **筋肉マップビジュアル**: SVGベースの21筋肉リアルタイム表示は競合にない独自性
2. **感情ドリブンのゴール設定**: 「舐められたくない」「モテたい」は日本市場のフィットネス初心者に刺さる
3. **ルーティンのパーソナライズ深度**: 目標×頻度×場所×経験×体重の5変数でルーティンを自動生成

### MuscleMapの競合劣位

1. **オンボーディングの長さ**: 競合の2-3倍のページ数。最短のHevyは2ページ
2. **ハードPaywallの攻撃性**: 閉じるボタンなしは競合で唯一（Fitbodもハードだが閉じるボタンあり）
3. **社会的証明の欠如**: レビュー数、ダウンロード数、ユーザーの声が一切ない
4. **入力摩擦の高さ**: 身長+体重+ニックネームの3入力は競合最多

## 10. 結論

MuscleMapのオンボーディングは**感情設計とパーソナライズ深度では競合を大幅に上回る**が、**ページ数の多さ（9ページ）と入力摩擦（特にPage 4, 6）が完走率を35-45%に押し下げている**。

最も投資対効果が高い施策は:
1. **PaywallViewの閉じる導線追加**（P0: 審査ブロッカー解消）
2. **WeightInputPageの入力項目削減**（P1: 離脱率20%→8%）
3. **Paywall への社会的証明追加**（P1: Trial開始率15%→25%）

これらの3施策だけで、MRRは現状の¥10,200から¥20,000-26,000への倍増が見込める。
